# 养老机构预收费资金监管平台 - 全部功能列表

## 一、系统概述

### 1.1 系统基本信息

- **系统名称**: 养老机构预收费资金监管平台
- **系统版本**: v1.0
- **开发框架**: 若依(RuoYi) v3.9.0
- **系统定位**: 养老机构预收费资金全流程监管,保障老年人权益
- **架构模式**: 四端协同(民政监管端 + 养老机构端 + H5小程序端 + 数据统计平台)
- **开发策略**: 单模块开发,功能导向,快速迭代

### 1.2 系统目标

本系统旨在通过信息化手段,实现对养老机构预收费资金的全流程监管,包括:
- 机构入驻审批和资质管理
- 老人入住和账户管理
- 订单支付和资金监管
- 押金使用和资金划拨
- 预警监控和风险防控
- 数据统计和决策支持

### 1.3 用户角色

- **系统管理员**: 系统配置、用户管理、权限分配
- **民政监管人员**: 机构审批、资金监管、预警核验
- **机构管理员**: 机构运营、入住管理、订单处理
- **机构工作人员**: 日常业务操作
- **老人家属**: H5端查询、支付、申请
- **老人**: H5端查看信息

##### 2.2.4.2 新增入住 (menu_id: 2032)
**功能描述**: 办理新入住手续。

**权限按钮**:
- 入住新增 (pension:checkin:add)

**主要功能**:
- 步骤1: 录入老人基本信息
  - 姓名、性别、出生日期、身份证号
  - 联系电话、紧急联系人
  - 健康状况、既往病史
  - 护理等级(自理、半护理、全护理)
  - 上传老人照片
  - 上传身份证照片
- 步骤2: 录入家属信息
  - 家属姓名、关系、联系电话
  - 关联用户账号(用于H5端登录)
- 步骤3: 选择床位
  - 选择房间号、床位号
  - 显示床位价格、护理价格
  - 选择餐费档次
- 步骤4: 生成入驻订单
  - 选择缴费周期(1个月、3个月、6个月、12个月)
  - 自动计算订单金额
    - 床位费 = 床位价格 × 月数
    - 护理费 = 护理价格 × 月数
    - 餐费 = 餐费价格 × 月数
    - 押金(一次性)
    - 会员费(一次性)
  - 生成订单明细
- 步骤5: 提交审批
  - 提交入住申请
  - 等待审批

**业务流程**:
1. 机构工作人员录入老人信息
2. 录入家属信息
3. 选择床位
4. 生成入驻订单
5. 提交审批
6. 审批通过后,老人状态变更为"已入住"
7. 床位状态变更为"占用"
8. 家属在H5端支付订单
9. 支付成功后,自动创建老人账户
10. 根据订单明细分配账户余额

## 四、核心业务流程说明

### 4.1 机构入驻流程

**流程步骤**:
1. 机构提交入驻申请
   - 填写机构基本信息(机构名称、信用代码、负责人、联系方式等)
   - 上传资质材料(营业执照、批准证书、监管协议)
2. 民政部门审批
   - 查看机构申请详情
   - 审核资质材料
   - 审批通过或驳回
3. 审批通过后
   - 自动生成机构管理员账号
   - 机构状态变更为"已入驻"(status = 1)
   - 发送通知给机构
4. 机构完善公示信息
   - 编辑机构简介
   - 上传环境照片
   - 配置服务项目和收费标准

**涉及数据表**:
- pension_institution: 机构基本信息
- pension_institution_attach: 机构附件材料
- sys_user: 机构管理员账号
- sys_user_institution: 用户机构关联

**关键控制器**: `InstitutionManageController.java`

---

### 4.2 老人入住流程

**流程步骤**:
1. 录入老人基本信息
   - 姓名、身份证、年龄、健康状况、护理等级
2. 录入家属信息
   - 通过elder_family表关联用户账号
3. 提交入住申请
   - 选择床位、护理等级、缴费周期
4. 生成入驻订单
   - 包含床位费、护理费、餐费、押金、会员费等明细
5. 审批通过
   - 分配床位
   - 创建账户
   - 更新床位状态为"占用"
6. 家属在H5端支付订单
7. 支付成功后
   - 更新订单状态为"已支付"
   - 创建或更新老人账户
   - 根据订单明细分配账户余额
   - 入驻订单自动扣除首月服务费

**涉及数据表**:
- elder_info: 老人基本信息
- elder_family: 老人家属关联
- elder_check_in: 入住申请
- bed_info: 床位信息
- order_info: 订单主表
- order_item: 订单明细
- account_info: 老人账户

**关键控制器**: `ElderCheckInController.java`

---

### 4.3 订单支付流程

**流程步骤**:
1. 用户在H5端选择订单支付
   - 验证用户权限(通过elder_family表)
2. 点击"立即支付"按钮
   - 显示确认对话框(订单号+金额)
3. 调用支付接口 `/h5/payment/process/{orderId}`
   - 模拟支付处理(1秒延迟)
4. 支付成功后执行以下操作:
   - 更新订单状态: 待支付(0) → 已支付(1)
   - 创建支付记录 → payment_record表
   - 记录监管账户流水 → supervision_account_log表(收入)
   - 更新老人账户余额 → 根据订单明细分配到不同余额类型

**账户余额分配逻辑** (updateAccountBalanceFromOrder方法):
```
根据order_item的item_type分配:
- deposit → deposit_balance (押金余额,不可用于消费)
- member_fee → member_balance (会员费余额)
- bed_fee, care_fee等 → service_balance (服务费余额,可用于消费)

入驻订单特殊处理:
- 支付成功后自动扣除首月服务费(床位费+护理费)
- 从service_balance中扣除
- 生成监管账户支出流水
```

5. 返回支付成功结果
6. 前端刷新订单列表

**涉及数据表**:
- order_info: 订单主表
- order_item: 订单明细
- payment_record: 支付记录
- account_info: 老人账户
- supervision_account_log: 监管账户流水

**关键控制器**: `H5OrderController.java` (第956-1094行: 支付处理, 第1099-1180行: 账户余额更新)

---

### 4.4 押金管理流程

**流程步骤**:
1. 机构发起押金使用申请
   - 填写申请原因、金额、用途
   - 上传相关凭证
2. 家属审批
   - 家属在H5端查看申请详情
   - 同意或拒绝申请
3. 家属审批通过后,提交给监管部门
4. 监管部门审批
   - 审核申请合理性
   - 同意或拒绝申请
5. 审批通过后
   - 从押金余额划拨到基本账户
   - 生成监管账户支出流水
   - 记录到supervision_account_log表

**申请状态流转**:
```
draft (草稿)
→ pending_family (待家属审批)
→ family_approved (家属已审批)
→ pending_supervision (待监管审批)
→ approved (已通过) / rejected (已驳回)
```

**涉及数据表**:
- deposit_apply: 押金申请
- account_info: 老人账户
- supervision_account_log: 监管账户流水

**关键控制器**: `DepositApplyController.java` (第268-275行: 押金划拨流水记录)

### 4.6 预警监控流程

**流程步骤**:
1. 系统定时任务检查预警规则
   - 费用超额预警
   - 押金超额预警
   - 入驻超额预警
   - 监管超额预警
   - 风险保证金预警
   - 大额支付预警
   - 突发风险预警
2. 触发预警后
   - 生成预警记录 → supervision_warning表
   - 发送通知给相关人员
3. 监管人员处理预警
   - 查看预警详情
   - 填写处理措施
   - 标记已处理或已忽略

**预警等级**:
- 1: 提示
- 2: 警告
- 3: 严重

**涉及数据表**:
- supervision_warning: 监管预警
- warning_rule_config: 预警规则配置

**关键控制器**: `WarningController.java`, `WarningRuleConfigController.java`

## 五、权限管理说明

### 5.1 权限控制模型

本系统采用基于角色的访问控制(RBAC)模型,实现菜单权限和按钮权限的细粒度控制。

**权限层级**:
1. **菜单权限**: 控制用户可以访问哪些页面
2. **按钮权限**: 控制用户可以执行哪些操作
3. **数据权限**: 控制用户可以查看哪些数据

### 5.2 数据权限控制

**机构管理员**:
- 只能查看和操作本机构的数据
- 通过sys_user_institution表关联机构
- 数据过滤: institution_id = currentUserInstitutionId

**家属**:
- 只能查看和操作关联老人的数据
- 通过elder_family表关联老人
- 数据过滤: elder_id IN (SELECT elder_id FROM elder_family WHERE user_id = currentUserId)

**民政监管人员**:
- 可以查看所有数据
- 无数据过滤限制

### 5.3 权限配置规范

**核心原则**: 功能页面必须包含该功能所需的所有权限按钮

**设计目标**:
- 方便在系统管理-角色管理中集中勾选权限
- 避免权限分散在多个菜单下,导致遗漏
- 机构管理员只需在养老机构导航下配置权限,不需要看到其他主导航菜单

**实施规范**:

每个功能页面(menu_type='C')下必须包含该页面所需的所有权限按钮(menu_type='F'),包括:
- 基础权限: 查询(query)、新增(add)、修改(edit)、删除(remove)、导出(export)
- 业务权限: 支付(pay)、取消(cancel)、审核(approve)等业务操作
- **关联数据权限**: 如果页面需要查询其他表的数据,也要添加对应的权限按钮

**示例: 订单列表页面(menu_id=2041)的权限配置**:
```
订单列表 (order:info:list) - 菜单页面
├── 订单列表权限 (order:info:list) - 列表查询
├── 订单列表查询 (order:info:query) - 详情查询
├── 订单列表新增 (order:info:add)
├── 订单列表修改 (order:info:edit)
├── 订单列表删除 (order:info:remove)
├── 订单列表导出 (order:info:export)
├── 订单支付 (order:info:pay)
├── 订单取消 (order:info:cancel)
├── 生成订单 (order:info:generate)
├── 订单明细查询 (order:item:query) ⭐ 关联数据权限
└── 支付记录查询 (payment:record:list) ⭐ 关联数据权限
```

**新功能开发检查清单**:
- [ ] 创建菜单页面(menu_type='C')
- [ ] 为该页面添加所有需要的权限按钮(menu_type='F')
  - [ ] 基础CRUD权限
  - [ ] 业务操作权限
  - [ ] 关联数据查询权限(分析页面会调用哪些API)
- [ ] 在控制器方法上添加 `@PreAuthorize` 注解
- [ ] 在前端页面按钮上添加 `v-hasPermi` 指令
- [ ] 为默认角色分配必要的权限(如机构管理员role_id=100)

**权限命名规范**:
- 格式: `模块:功能:操作`
- 示例:
  - `order:info:list` - 订单主表列表
  - `order:info:query` - 订单主表查询
  - `order:item:query` - 订单明细查询
  - `payment:record:list` - 支付记录列表

### 5.4 前后端权限控制

**后端权限控制**:
```java

@GetMapping("/list")

    // 业务逻辑
}
```

## 八、总结

本系统是一个功能完整的**四端协同养老机构预收费资金监管平台**:

1. **管理端(PC)**: 系统管理、养老机构管理、民政监管、数据统计
2. **机构端(PC)**: 机构日常运营管理(入住、订单、押金、资金划拨等)
3. **监管端(PC)**: 民政部门监管功能(机构审批、预警核验、资金监管等)
4. **用户端(H5)**: 家属和老人使用(机构浏览、预约、订单、费用查询等)

**核心业务流程**:
机构入驻 → 审批 → 床位管理 → 老人入住 → 订单支付 → 账户管理 → 押金使用 → 资金划拨 → 预警监控 → 数据统计

**后端控制器**: 80+个,覆盖所有业务功能

**前端页面组件**: 150+个(PC端 + H5端)

**权限管理**: 基于RBAC,支持菜单权限、按钮权限、数据权限
